## 🌿 分支策略

本项目采用 **Git Flow** 分支管理策略，确保代码质量和版本控制的规范性。

### 分支类型

| 分支类型 | 命名规则 | 说明 | 生命周期 | 来源 | 目标 |
|---------|---------|------|---------|------|------|
| **main** | `main` | 生产环境稳定版本 | 永久 | - | - |
| **dev** | `dev` | 开发集成分支 | 永久 | main | main |
| **feature** | `feature/*` | 新功能开发 | 临时 | dev | dev |
| **fix** | `fix/*` | Bug 修复 | 临时 | dev | dev |
# 分支详细说明

#### 1. main 分支（主分支）

- **用途**：存放生产环境的稳定代码
- **保护规则**：
    - ❌ 禁止直接推送代码
    - ✅ 只能通过 Pull Request 从 dev 或 hotfix 合并
    - ✅ 每次合并必须通过 CI 检查
    - ✅ 需要至少 1 人 Code Review

~~~
查看 main 分支
git checkout main 
git pull origin main
~~~


#### 2. dev 分支（开发分支）

- **用途**：日常开发集成，包含最新的功能
- **保护规则**：
  - ❌ 禁止直接推送（建议）
  - ✅ 从 feature/fix 分支合并
  - ✅ 需要通过 CI 检查

~~~
切换到 dev 分支
git checkout dev 
git pull origin dev
~~~
#### 3. feature 分支（功能分支）

- **用途**：开发新功能
- **命名规范**：`feature/功能描述`
- **示例**：
  - `feature/user-authentication`
  - `feature/course-management`
  - `feature/file-upload`

~~~
创建功能分支
git checkout dev 
git pull origin dev 
git checkout -b feature/user-authentication（以具体开发功能命名的分支）
开发完成后合并回 dev
git checkout dev 
git merge feature/user-authentication 
git push origin dev
删除已合并的功能分支
git branch -d feature/user-authentication 
git push origin --delete feature/user-authentication
~~~


#### 4. fix 分支（修复分支）

- **用途**：修复非紧急 Bug
- **命名规范**：`fix/问题描述`
- **示例**：
  - `fix/login-validation`
  - `fix/database-connection`


## 开发流程

### 标准流程

~~~
创建分支 → 本地开发 → 提交代码 → 推送远程 
→ 创建 PR → Code Review → CI 检查 → 合并 → 删除分支
~~~


### 快速开始

~~~
1. 创建功能分支
git checkout dev
git pull origin dev
git checkout -b feature/功能名称

2. 开发并提交
git add . git commit -m "feat: 功能描述"

3. 推送到远程
git push origin feature/功能名称

4. 创建 Pull Request（GitHub/Gitee 网页操作）

5. 合并后清理
git checkout dev 
git merge feature/功能名称 
git branch -d feature/功能名称 
git push origin --delete feature/功能名称
~~~


### 日常习惯

- **开始前**：拉取最新代码 `git pull origin dev`
- **工作中**：频繁小步提交，保持原子性
- **完成后**：及时创建 PR，避免长期分支
- **合并后**：立即删除已合并分支

### 注意事项

⚠️ **禁止**直接推送到 `main` 和 `dev` 分支  
⚠️ **必须**通过 Pull Request 合并代码  
⚠️ **确保** CI 检查通过后再合并  
⚠️ **及时**处理 Review 意见


## 代码检查指南

### 后端检查（Java）

#### 编译与测试

~~~
编译检查
mvn clean compile
运行测试
mvn test
代码风格检查
mvn checkstyle:check
~~~

#### 重点检查项
- [ ] Lombok 注解正确使用
- [ ] JPA 实体映射正确
- [ ] 异常统一处理
- [ ] 日志记录完整
- [ ] 无硬编码配置

---

### 前端检查（Vue3）

####  lint 与构建

~~~
bash
 cd frontend
代码检查
npm run lint
构建测试
npm run build
~~~

#### 重点检查项
- [ ] Composition API 规范
- [ ] Props 完整定义
- [ ] 响应式数据正确使用
- [ ] 组件命名符合规范
- [ ] 无控制台警告

---

### 通用检查

#### 提交前
- [ ] 代码能正常编译/构建
- [ ] 单元测试通过
- [ ] 无 ESLint/Checkstyle 错误
- [ ] 删除调试代码
- [ ] 格式化代码

#### 自动化检查
CI 流水线：
- ✅ 编译验证
- ✅ 单元测试
- ✅ 代码风格
- ✅ 构建打包

建议使用 Pull Request 进行代码检查
~~~
如何创建 Pull Request？

步骤 1：推送分支到远程
git push origin feature/xxx

步骤 2：在 GitHub 网页操作
-进入仓库页面
-点击 "Pull requests" 标签
-点击绿色按钮 "New pull request"
-选择分支：
   base: dev（要合并到的目标分支）
   compare: feature/user-login（你的功能分支）
-填写信息：
  标题：简洁描述改动
  描述：详细说明改动内容、相关 Issue
-点击 "Create pull request"

步骤 3：等待审查和合并
-团队成员 review 代码
-CI 自动运行测试
-通过后点击 "Merge pull request" 按钮
-删除已合并的分支

~~~

**原则：本地检查通过后再提交，CI 全绿后再合并** ✓
