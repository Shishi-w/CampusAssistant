# BDD 用户故事 MoSCoW 优先级排序

基于《校园互助服务平台需求规格说明书（SRS v2.1）》及用例图（use-case-diagram.md），对 BDD 格式的 15 条用户故事进行 MoSCoW 优先级排序。

**MoSCoW 定义**：

- **Must (M)**：必须实现。缺少则核心业务流程断裂，MVP 无法交付。
- **Should (S)**：应该实现。缺失会显著降低用户体验，但可通过简化方案或替代流程临时支撑。
- **Could (C)**：可以实现。有则提升体验，无则不影响 MVP 基础运转，资源紧张时可延后。
- **Won't (W)**：本次不做。明确放到后续版本。

---

## 一、Must（必须实现）— 核心交易闭环 + 信任兜底

以下 6 条用户故事构成平台 MVP 的骨架，缺一则互助交易无法完整运转或纠纷无法解决。

### M1. US-01 学号认证登录

**关联用例**：UC01（SSO学号认证登录）
**优先级**：P0

> **作为** 一名在校学生，
> **我想要** 通过学号统一认证登录平台，
> **以便** 确保平台仅限校内真实身份的用户使用。

**场景：首次登录自动注册**

```gherkin
Given 用户尚未在平台注册
When 用户在登录页输入正确的学号和SSO密码并提交
Then 系统调用校园SSO接口验证身份
And 验证通过后自动创建平台账户并绑定学号
And 返回用户会话令牌，跳转至首页
```

**场景：登录失败提示**

```gherkin
Given 用户在登录页
When 用户输入错误的学号或密码并提交
Then 系统显示错误提示"学号或密码错误"
And 不创建会话令牌
And 保留用户已输入的学号以便重新尝试
```

### M2. US-02 发布互助需求

**关联用例**：UC05（发布互助需求）
**优先级**：P0

> **作为** 一名有互助需求的学生（求助方），
> **我想要** 发布一条包含分类、地点、报酬的互助需求，
> **以便** 其他校内同学可以看到并接单帮助我。

**场景：成功发布积分悬赏需求**

```gherkin
Given 用户已登录且积分余额充足
When 用户在发布页填写标题"代取快递"、分类"快递代取"、地点"仙林校区-菜鸟驿站"、报酬类型"积分悬赏"、积分数额"20"
And 点击"发布"按钮
Then 系统校验标题长度（2-50字）和积分余额
And 生成状态为"待接单"的需求订单
And 需求出现在需求广场列表中
And 用户收到"发布成功"提示
```

**场景：发布时积分余额不足**

```gherkin
Given 用户已登录且当前积分余额为10
When 用户设置积分悬赏为20并尝试发布
Then 系统提示"积分余额不足，请调整悬赏金额或选择无偿帮助"
And 不生成需求订单
```

### M3. US-06 接单

**关联用例**：UC10（接单）
**优先级**：P0

> **作为** 一名帮助方，
> **我想要** 浏览需求列表并选择接单，
> **以便** 确认承接该互助任务。

**场景：成功接单**

```gherkin
Given 用户已登录且某需求状态为"待接单"
When 用户点击"接单"按钮
Then 系统检查该需求当前状态
And 使用数据库锁防止并发冲突
And 将需求状态更新为"进行中"
And 记录帮助方用户ID
And 向求助方发送"您的需求已被接单"站内通知
And 向帮助方展示"接单成功"提示
```

**场景：已被他人接单**

```gherkin
Given 用户正在查看某需求详情
When 该需求在页面加载后已被其他用户接单（状态变为"进行中"）
And 用户点击"接单"按钮
Then 系统提示"该需求已被其他用户接单"
And 不更新订单状态
```

---

### M4. US-10 确认完成与报酬转移

**关联用例**：UC12（确认完成与报酬转移）
**优先级**：P0

> **作为** 一名求助方，
> **我想要** 在确认任务完成后让系统自动转移积分报酬，
> **以便** 帮助方及时获得应有的奖励。

**场景：正常确认完成**

```gherkin
Given 帮助方已提交完成凭证，订单状态为"待确认"
When 求助方在订单详情页点击"确认完成"
Then 系统将订单状态更新为"已完成"
And 自动将悬赏积分从求助方账户划转至帮助方账户（事务性操作）
And 触发双向匿名评价流程
And 向双方发送"订单已完成，请互相评价"通知
```

---

### M5. US-12 发起争议仲裁

**关联用例**：UC13（发起争议仲裁）
**优先级**：P0

> **作为** 一名对订单结果有异议的用户，
> **我想要** 向平台发起争议申诉并提交证据，
> **以便** 请求管理员介入裁决。

**场景：发起争议**

```gherkin
Given 订单状态为"待确认"或"已完成"
When 用户点击"发起争议"并填写争议原因"任务未实际完成"，同时勾选证据类型"聊天记录"和"照片"
Then 系统将订单状态冻结为"争议中"
And 生成仲裁工单并推送至管理员后台
And 阻止积分自动划转或撤回已划转的积分
And 向双方发送"争议已提交，等待平台处理"通知
```

### M6. US-15 管理员处理争议仲裁

**关联用例**：UC23（争议仲裁处理）
**优先级**：P0

> **作为** 一名平台管理员，
> **我想要** 查看争议双方提交的证据并做出仲裁判定，
> **以便** 维护平台秩序并公正解决纠纷。

**场景：查看证据并判定求助方胜诉**

```gherkin
Given 管理员已登录后台且存在一条待处理的争议工单
When 管理员在仲裁页面查看争议详情，包括：双方聊天记录、帮助方上传的完成凭证照片、地点签到记录、时间线
And 管理员判定帮助方未实际完成任务
And 点击"判定求助方胜诉"
Then 系统将订单状态更新为"已关闭-求助方胜诉"
And 若积分已划转，则将积分退回求助方账户
And 帮助方信用分扣减10分
And 向双方发送仲裁结果通知
And 工单状态更新为"已处理"
```

**场景：查看证据后判定帮助方胜诉**

```gherkin
Given 管理员已登录后台且存在一条待处理的争议工单
When 管理员查看证据后判定帮助方已完成任务且求助方恶意不确认
And 点击"判定帮助方胜诉"
Then 系统将订单状态更新为"已完成-帮助方胜诉"
And 积分划转至帮助方（若尚未划转）
And 求助方信用分扣减5分
And 向双方发送仲裁结果通知
```

## 二、Should（应该实现）— 交易支撑与信任体系

以下 6 条用户故事对用户体验和平台可信度至关重要，但在资源极度紧张时可通过简化方案临时支撑。

### S1. US-04 浏览并筛选需求列表

**关联用例**：UC07（浏览与筛选需求列表）
**优先级**：P0

> **作为** 一名想要提供帮助的学生（帮助方），
> **我想要** 按分类、距离、发布时间筛选需求，
> **以便** 快速找到我能够且愿意接手的任务。

**场景：按分类和距离筛选**

```gherkin
Given 用户已登录并位于需求广场页面
When 用户选择筛选条件：分类="快递代取"，排序="距离最近"
Then 系统仅展示分类为"快递代取"的需求
And 按需求地点与用户当前位置的距离由近到远排序
And 每条需求展示标题、模糊位置、悬赏积分、发布时间
```

**场景：无筛选结果**

```gherkin
Given 用户已登录
When 用户选择分类="活动组队"且当前该分类下无待接单需求
Then 系统展示空白状态提示"暂无该类需求，试试其他分类吧"
```

---

### S2. US-05 查看需求详情

**关联用例**：UC09（查看需求详情）
**优先级**：P0

> **作为** 一名帮助方，
> **我想要** 在接单前查看需求的详细信息和发布者的信用情况，
> **以便** 判断是否适合接单并评估风险。

**场景：查看需求详情页**

```gherkin
Given 用户在需求广场看到一条"代取快递"需求
When 用户点击该需求卡片
Then 系统展示需求详情：标题、描述、分类、模糊位置（"仙林校区-图书馆附近"）、期望完成时间、报酬类型与数额
And 展示发布者信用等级（如"良好"）和历史好评率
And 不展示发布者的精确坐标和真实姓名
```

### S3. US-09 提交完成凭证

**关联用例**：UC11（提交完成凭证）
**优先级**：P0

> **作为** 一名帮助方，
> **我想要** 完成任务后上传照片等凭证，
> **以便** 向求助方证明任务已按要求完成。

**场景：上传完成凭证**

```gherkin
Given 用户已承接某需求且任务已实际完成
When 用户在订单详情页点击"提交凭证"并上传2张任务完成照片（每张<5MB）
Then 系统对图片进行压缩和敏感内容检测
And 将凭证关联至该订单
And 将订单状态更新为"待确认"
And 向求助方发送"对方已提交完成凭证，请确认"通知
```

**场景：上传图片超过数量限制**

```gherkin
Given 用户正在提交完成凭证
When 用户尝试上传第4张图片
Then 系统提示"最多上传3张凭证照片"
And 阻止继续上传
```

### S4. US-08 接单后私聊沟通

**关联用例**：UC16（接单后私聊沟通）
**优先级**：P0

> **作为** 一名已接单的帮助方，
> **我想要** 与求助方进行实时站内私聊，
> **以便** 沟通任务细节、约定见面地点和时间。

**场景：正常私聊**

```gherkin
Given 用户已承接某需求（订单状态为"进行中"）
When 用户在订单详情页进入"私信"页面并发送消息"我在图书馆门口等你"
Then 系统通过WebSocket实时推送消息给对方
And 聊天记录持久化存储，作为争议仲裁的证据来源
And 消息展示发送时间和发送者匿名昵称
```

**场景：订单完成后私聊自动关闭**

```gherkin
Given 某订单状态已变为"已完成"
When 帮助方尝试在订单私信页面发送新消息
Then 系统提示"订单已完成，私信通道已关闭"
And 不发送该消息
```

### S5. US-11 双向匿名评价

**关联用例**：UC14（双向匿名评价）
**优先级**：P0

> **作为** 一名用户，
> **我想要** 在订单完成后对对方进行匿名评分和评价，
> **以便** 为平台的信用体系提供反馈。

**场景：提交匿名评价**

```gherkin
Given 订单状态为"已完成"且用户尚未评价
When 用户填写评分（1-5星）和评价内容"对方很准时，谢谢！"
And 点击"提交评价"
Then 系统记录评价并关联至该订单
And 重新计算双方信用分（好评+2分）
And 评价展示为匿名形式（如"帮助者_A3F2"）
And 评价提交后48小时内允许修改一次
```

### S6. US-13 获取积分奖励

**关联用例**：UC19（获取积分奖励）
**优先级**：P0

> **作为** 一名帮助方，
> **我想要** 在成功完成互助任务后获得积分奖励，
> **以便** 积累积分用于后续兑换商品或发布需求。

**场景：完成订单后获得积分**

```gherkin
Given 帮助方已完成某积分悬赏订单且求助方已确认完成
When 系统自动执行积分划转
Then 帮助方账户增加对应积分数额
And 系统生成积分流水记录（来源：完成任务，订单号：XXX）
And 帮助方收到"您已获得20积分"通知
And 用户可在"个人中心-积分明细"中查看该记录
```

## 三、Could（可以实现）— 锦上添花

以下 3 条用户故事能够提升体验，但缺失不影响 MVP 作为互助平台的基础运转。

### C1. US-03 为需求添加标签

**关联用例**：UC06（添加需求标签）
**优先级**：P1

> **作为** 一名求助方，
> **我想要** 在发布需求时添加标签（如"急单""需排队"），
> **以便** 帮助方能更快地识别需求的紧急程度和特征。

**场景：发布时添加标签**

```gherkin
Given 用户正在发布互助需求
When 用户在标签栏输入"#急单#"和"#菜鸟驿站#"（共2个标签）
And 完成其他必填项并发布
Then 系统将标签与需求关联并建立索引
And 需求详情页展示这两个标签
And 其他用户可通过标签关键词搜索到该需求
```

**场景：标签超出数量限制**

```gherkin
Given 用户正在发布互助需求
When 用户尝试添加第4个标签
Then 系统提示"最多添加3个标签"
And 阻止继续添加
```

### C2. US-07 向陌生人打招呼

**关联用例**：UC15（向陌生人打招呼）
**优先级**：P0

> **作为** 一名帮助方，
> **我想要** 在接单前向需求发布者发送一条预沟通消息，
> **以便** 确认细节或判断自己是否适合接单。

**场景：首次打招呼**

```gherkin
Given 用户已登录且未承接该需求
When 用户在需求详情页点击"打招呼"并发送消息"请问快递大概什么时候到？"
Then 系统向求助方发送该消息
And 记录该对话为"预沟通"状态
And 限制帮助方在对方回复前仅可发送这一条消息
```

**场景：对方未回复时再次尝试打招呼**

```gherkin
Given 用户已向某求助方发送过打招呼消息且对方未回复
When 用户再次尝试向同一求助方发送打招呼消息
Then 系统提示"对方回复前无法再次发送消息"
And 不发送新消息
```

### C3. US-14 积分兑换商品

**关联用例**：UC20（积分兑换商品）
**优先级**：P0

> **作为** 一名积累了积分的用户，
> **我想要** 使用积分兑换平台提供的商品或服务，
> **以便** 将虚拟积分转化为实际价值。

**场景：成功兑换商品**

```gherkin
Given 用户已登录且当前积分余额为200
When 用户在积分商城选择"小黄车月卡（150积分）"并点击"兑换"
Then 系统校验积分余额（200 ≥ 150）
And 扣除150积分（事务性操作）
And 生成兑换订单并展示兑换码
And 向用户发送"兑换成功"通知
```

**场景：积分不足无法兑换**

```gherkin
Given 用户当前积分余额为100
When 用户尝试兑换"小黄车月卡（150积分）"
Then 系统提示"积分余额不足，当前余额100，需150"
And 不扣除积分，不生成兑换订单
```

## 四、Won't（本次不做）

本次 15 条用户故事均在合理范围内，**Won't 列表为空**。以下功能明确超出本期范围，已在 SRS 第 2.7 节列为 P1/P2：

| 功能                     | 排除原因                      | 计划版本 |
| ------------------------ | ----------------------------- | -------- |
| 移动端微信小程序/H5 适配 | 超出当前 MVP 人力与开发周期   | P1       |
| 按距离排序推荐           | 需要地图数据精度保障，P2 评估 | P2       |
| 周期用户调研反馈入口     | 运营功能，非技术阻塞          | P2       |
| 数据统计看板扩展         | 管理后台增强，P1 完善         | P1       |
