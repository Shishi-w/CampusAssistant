

### 用例1：发布需求

| 字段 | 内容 |
| --- | --- |
| 用例名称 | 发布需求 |
| 参与者 | 求助者 |
| 前置条件 | 用户信息可识别且用户状态正常 |
| 后置条件 | 需求状态为“已发布”，其他用户可浏览和接单该需求 |

#### 基本流
| 步骤 | 行为描述 |
| --- | --- |
| 1 | 用户点击“发布需求”按钮 |
| 2 | 系统显示发布需求表单 |
| 3 | 用户填写需求标题 |
| 4 | 用户选择需求分类（快递代取、学习辅导、二手交易、活动组队、失物招领、咨询、其他） |
| 5 | 用户填写详细描述 |
| 8 | 用户设置报酬类型（积分/现金/无偿）及具体金额 |
| 9 | 用户添加相关标签（如#急单# #需排队#） |
| 11 | 用户点击“发布”按钮 |
| 12 | 系统进行审核 |
| 13 | 审核通过后，需求发布成功，系统显示发布成功提示 |

#### 替代流
| 编号 | 扩展条件 | 处理方式 |
| --- | --- | --- |
| 3a | 非法标题 | 系统拒绝输入并提示用户修改 |
| 5a | 描述字数超出上限 | 系统拒绝输入并提示用户修改 |
| 12a | 审核不通过 | 系统向用户发送审核不通过通知 |
| 13a | 发布后编辑 | 用户在需求未接单前点击“编辑”按钮，修改需求信息，重新发布 |
| 13b | 发布后撤回 | 用户在需求未接单前点击“撤销”按钮，将需求状态改为“已撤回” |

### 用例2：接单流程

| 字段 | 内容 |
| --- | --- |
| 用例名称 | 接单流程 |
| 参与者 | 帮助方、求助方 |
| 前置条件 | 需求订单状态为“已发布”且未被接单；帮助方已登录平台且账号状态正常 |
| 后置条件 | 任务状态更新为“已完成”；帮助方获得相应报酬；双方信用分已更新；订单记录已保存到系统 |

#### 基本流
| 步骤 | 行为描述 |
| --- | --- |
| 1 | 帮助方浏览需求列表或搜索特定需求 |
| 2 | 帮助方查看需求详情，确认自己可以完成 |
| 3 | 帮助方点击“申请接单”按钮 |
| 4 | 跳转聊天界面，求助方与帮助方沟通细节，求助方可选择是否同意接单 |
| 5 | 同意接单后，系统将需求状态改为“已接单” |
| 6 | 帮助方开始执行任务 |
| 7 | 任务完成后，帮助方点击“完成任务”按钮 |
| 8 | 帮助方上传完成凭证（照片、文字说明） |
| 9 | 系统通知求助方任务已完成，等待确认 |
| 10 | 求助方查看完成凭证，确认任务完成 |
| 11 | 系统自动将报酬（积分/现金托管）转给帮助方 |
| 12 | 帮助方确认收到报酬 |
| 13 | 系统将订单状态改为“已完成” |
| 14 | 系统更新双方信用分 |
| 15 | 系统提示双方进行互评 |

#### 替代流
| 编号 | 扩展条件 | 处理方式 |
| --- | --- | --- |
| 5a | 帮助方拒绝接单请求 | 系统向帮助方通知并禁止该帮助方对该需求进行接单申请 |
| 10a | 求助方超时未确认 | 系统自动确认任务完成 |
| 12a | 帮助方超时未确认 | 系统自动确认收到报酬 |
| 10-12a | 任务争议 | 用户提交争议原因，系统将订单状态改为“争议中”；管理员介入仲裁；根据仲裁结果处理订单 |

### 用例3：争议仲裁

| 字段 | 内容 |
| --- | --- |
| 用例名称 | 争议仲裁 |
| 参与者 | 求助方、帮助方、管理员 |
| 前置条件 | 订单状态为“已接单”或“待确认”；争议已由任一方发起，系统记录争议原因及凭证 |
| 后置条件 | 根据仲裁结果更新订单状态；系统处理托管的报酬；调整双方信用分；争议记录保存至系统 |

#### 基本流
| 步骤 | 行为描述 |
| --- | --- |
| 1 | 求助方或帮助方在订单详情页点击“发起争议”按钮 |
| 2 | 系统显示争议表单，要求填写争议原因（如任务未完成、凭证造假等）并上传证据（截图、聊天记录等） |
| 3 | 用户提交争议 |
| 4 | 系统将订单状态改为“争议中”，并通知管理员待处理 |
| 5 | 管理员在后台查看争议详情，包括订单信息、双方描述、证据材料 |
| 6 | 管理员根据平台规则进行判断，选择仲裁结果 |
| 7 | 系统执行仲裁结果：<br>  - 若支持求助方：订单取消，报酬退回求助方；帮助方信用分扣减<br>  - 若支持帮助方：订单完成，报酬转给帮助方；求助方信用分扣减<br> |
| 8 | 系统将仲裁结果通过站内信通知双方 |
| 9 | 订单状态更新 |

#### 替代流
| 编号 | 扩展条件 | 处理方式 |
| --- | --- | --- |
| 5-6a | 双方在仲裁前自行和解 | 任一方在管理员仲裁前点击“撤销争议”,系统要求另一方确认和解；系统将订单恢复原状态（“已接单”或“待确认”），争议记录关闭 |
| 5-6b | 管理员要求补充材料 | 管理员在仲裁时选择“需补充材料”，并说明要求；系统通知发起方补充材料；发起方重新提交证据，回到步骤6；若超时未补充，管理员可按现有材料裁决 |

## AI生成用例与人工对比
AI生成版在共同用例中遗漏了以下替代流：

* 发布需求：人工版包含“5a 描述字数超出上限”和“13b 发布后撤回”，AI版未提及。

* 接单流程：人工版包含“5a 帮助方拒绝接单请求”和“12a 帮助方超时未确认”。AI版因流程设计不同（直接接单无需对方同意，且报酬自动转移无需帮助方二次确认），前者自然不存在，后者对应步骤缺失。

其他主要差异可归纳为两类：
一、架构设计差异

* 用例粒度：人工版将接单流程合并为一个长流程（含申请、沟通、同意、执行、凭证、确认、报酬、评价共15步），AI版拆分为UC10、UC11、UC12等多个细粒度用例。    

二、内容细节差异  

* 数值约束：人工版未给出具体数值限制（如标题字数、描述上限），AI版包含具体阈值（如标题2-50字、描述≤500字）。

* 接单机制：人工版需要求助方同意接单，AI版直接接单。

* 报酬确认：人工版需要帮助方确认收到报酬，AI版自动转移。
